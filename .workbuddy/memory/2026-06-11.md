## TypeScript 类型基础
- 三个基本类型：string、number、boolean（小写，number 统包 int/float）
- `?` 可选属性：interface 里标记可以不传
- `|` 联合类型：`number | null` 表示可以是其中任意一种
- `[]` 数组类型：`string[]` = 字符串数组
- `(参数) => 返回值`：函数类型
- `void`：没有返回值
- interface：定义对象的形状（TypeScript），不等同于 Java 的 interface（定义行为）
- interface 可以嵌套：VideoHeaderProps 包含 videoInfo: VideoInfo

## 泛型与非空断言
- `useState<VideoInfo | null>` 是泛型，跟 Java `ArrayList<String>` 一个道理
- `!` 非空断言：骗过编译器，运行时可能炸（危险）
- `?.` 可选链：运行时安全访问，null/undefined 就短路返回 undefined（安全）
- 能用 `?.` 就不用 `!`

## typeof 与防御性编程
- `typeof` 是运行时运算符（JavaScript 原生），不是 TypeScript 编译时检查
- 后端返回的 JSON 没有类型信息，前端要层层检查才敢用
- 三元表达式 `条件 ? 真值 : 假值`：赋值场景用，执行操作用 if-else

## 客户端 vs 服务器
- `"use client"` 不是因为 JSX，而是因为 useState / onClick / useEffect
- 服务器也能渲染 JSX（Server Component）
- 判断标准：这个组件需要用户交互吗？

## VideoInfo 类型生命周期
- 定义：lib/types.ts → interface VideoInfo
- 后端组装：api/video-info/route.ts（没标注类型，信任区）
- 前端接收：useState<VideoInfo | null>(null) → setVideoInfo(data)（标注了类型，检查区）
- 传给组件：videoInfo! （非空断言）
- TypeScript 只在标注了类型的地方检查

## Supabase 数据库修复
- 问题：登录可用但收藏等功能失效
- 根因：整个数据库 schema 从未推送到 Supabase（所有表都不存在）
- `on_auth_user_created` 触发器在 migration 里也是注释掉的
- 修复：合并 16 个 migration 文件为 init-full.sql，修复 cron.schedule 语法错误（$$嵌套问题），在 Dashboard SQL Editor 一次性执行
- 步骤：DROP SCHEMA public CASCADE → CREATE SCHEMA public → 跑 init-full.sql
- 次要问题：SUPABASE_SERVICE_ROLE_KEY 已在 .env.local 中配置
- 额外修复：手动建表缺少 GRANT 权限，需执行 `GRANT ALL ON ALL TABLES IN SCHEMA public TO anon, authenticated;` 等
- 关键教训：Supabase Dashboard 建表会自动加权限，SQL 手动建表不会，必须手动 GRANT
