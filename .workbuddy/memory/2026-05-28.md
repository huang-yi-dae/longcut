# 2026-05-28

## Stage 5.7 — fetch + async/await

- fetch 是浏览器内置的网络请求函数，默认 GET
- async/await 配套使用，暂停等待异步操作完成
- useEffect 回调不能是 async（async 永远返回 Promise，React 要求 undefined 或清理函数）
- 正确写法：async 放在 useEffect 内层函数
- 回调 vs 普通返回：回调是把函数交给对方决定何时调用；类比"递信 vs 留电话"

## Stage 5.8 — try...catch + res.ok

- try...catch 类似 Python try...except
- fetch 特殊坑：404/500 catch 兜不住，必须手动 res.ok 检查
- HTTP 状态码：401 跳登录，其他 !res.ok 统一兜底

## Stage 5.9 — loading/error/data 三段式

- 三个 useState 管理请求的三种状态
- finally 确保 loading 一定关闭
- 组件重渲染机制：useState 变了 → React 自动重执行函数 → 重新 return

## Stage 5.10 — 自定义 Hook 封装 fetch

- useFetch：封装 fetch + loading + error 为可复用 Hook
- [url] 依赖数组：url 变了自动重新请求

## Stage 5.11 — useRef

- 返回 { current: null }，改 .current 不触发渲染
- ref 属性绑定 DOM 元素，React 自动塞进 .current
- useState vs useRef：需要页面更新用 useState，只想存值用 useRef

## Stage 5.12 — useReducer

- 复杂状态：发 action，reducer 决定如何改
- 返回 [state, dispatch] 数组
- 核心思路与 Zustand 相同：当前状态 + 指令 → 新状态

## Stage 6.1 — 表单处理

- 受控组件：useState + value + onChange
- e.target.value = 用户输入的内容
- 搜索 = useState + useDebounce + useFetch 组合

## Stage 6.2 — Server Actions

- 'use server' = 服务端函数，Next.js 自动生成隐藏 fetch
- Server Action vs route.ts：Action 适合表单写操作，route.ts 适合读数据/需要可控性/第三方调用
- 本质：fetch + route.ts 的框架层面封装

## Stage 6.3 — 认证与鉴权

- Authentication（认证）vs Authorization（鉴权）
- Token 机制：登录生成 → 前后端各存 → 每次请求验证
- HttpOnly Cookie 防 XSS 攻击

## Stage 6.4 — 数据库集成（Prisma ORM）

- Prisma = JS 版 SQL。schema.prisma 定义→push→generate→自动补全
- 多表查询：先声明关系，用 include 联查
- ORM 缺点：复杂查询别扭，生成 SQL 不一定最优（可回退 $queryRaw）

## Stage 6.5 — 部署上线

- git push 自动触发重新部署

## Stage 7.1 — 性能优化

- lazy + Suspense：代码分割，lazy 组件 throw Promise，Suspense 接住
