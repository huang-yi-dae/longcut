# 2026-06-16 学习日志

## 阶段 1 — Node.js 运行时深入（第 1~3 天）

### 1. 事件循环（Event Loop）
- 执行顺序：同步代码 → 微任务（Promise/await）→ 宏任务（setTimeout/setInterval）
- 微任务：Promise.then/catch/finally、async/await、process.nextTick（Node.js 独有，优先级最高）
- 宏任务：setTimeout、setInterval、setImmediate（Node.js 独有）、I/O 操作
- 判断标准：希望尽快执行用微任务，可以稍等等用宏任务
- async/await 不需要主动选，引擎自动放进微任务队列
- setTimeout(fn, 0) 不是真等 0ms，而是推到宏任务队列

### 2. async/await 执行机制
- await 暂停的是当前 async 函数，不阻塞外部代码
- Java 的 Thread.sleep 阻塞整个线程，JS 的 await 只暂停当前函数
- 练习题正确：1→4→6→3→5→2（同步→微任务→宏任务）

### 3. 模块系统（require vs import）
- CommonJS（require/module.exports）→ Node.js 老标准，运行时动态加载
- ES Module（import/export）→ ES6 新标准，编译时静态分析，LongCut 用的这种
- export default → 导入不加 {}，一个文件最多一个
- export 具名导出 → 导入加 {}，一个文件可以有多个
- from 后面跟的是文件路径/包名，跟第三方库还是自己写的无关

### 4. Stream（流）
- 核心：数据像水一样通过管道流过去，不用全部装进内存
- 不用流：fs.readFile 先把整个文件塞进内存 → 大文件会爆内存
- 用流：fs.createReadStream 每次只读一小块（默认 64KB）→ 内存始终很小
- 四种流：Readable（读）、Writable（写）、Duplex（又读又写）、Transform（读写+转换）
- .pipe() 连接两个流，自动搬运数据：createReadStream().pipe(createWriteStream())
- .on('data', chunk) 监听每读到一块数据，.on('end') 监听读完

### 5. Express + Stream 实战
- Express = Node.js 版 Flask，手动注册路由（app.get）
- stream.pipe(res) 等价于 Flask 的 return send_file()，流式返回大文件
- Express handler 由框架自动调用，不用手动调（跟 Flask @app.route 一样）
- res = 响应对象，等价于 Flask 的 return / Next.js 的 Response
- Flask 用 return 写响应，Express 直接操作 res 对象

### 6. API 安全五关
- 正确顺序：认证（你是谁）→ 限流（你能调吗）→ 输入校验（参数对吗）→ 鉴权（你能做吗）→ 输出过滤（返回数据安全吗）
- 认证 vs 鉴权：认证问"你是谁"，鉴权问"你凭什么做这个"
- Express 中间件：next() 表示放行，不调 next() 就拦截返回错误
- LongCut 的 middleware.ts 做的事跟 Express 中间件一样

### 7. Cluster 进程管理
- Node.js 单线程 = 1 个厨师轮流做几桌菜（事件循环切换任务）
- Cluster = 请多个厨师同时做（多进程并行利用多核 CPU）
- 主进程（isPrimary）负责分发请求，子进程负责处理业务
- 子进程不是真的监听端口，主进程监听端口后分发给子进程
- Cluster 子进程数 = CPU 核数（os.cpus().length）

### 8. child_process
- Node.js 调外部命令（Python/Go/Shell），开操作系统级别的子进程
- exec('python3 script.py') 像终端里敲命令，stdout/stderr 接收输出
- child_process 不关心语言，只关心启动命令和输出结果
- Cluster vs child_process：Cluster 开同类 Node.js 进程，child_process 开任意语言进程
- child_process 子进程数看内存，需自己控制上限；Cluster 子进程数看 CPU 核数

### 9. process.env
- 系统级别配置字典，等于 Python 的 os.environ
- 安全 + 灵活：密码不写在代码里，存在服务器环境变量中
- dotenv（Node.js）= load_dotenv（Python），从 .env 文件加载到 process.env
- Next.js 自动读 .env.local 并加载到 process.env

## 阶段 2 — MongoDB + Redis

### 10. MongoDB 存储
- PostgreSQL = 表格（Excel），MongoDB = 文档（JSON 对象）
- MongoDB 文档可以嵌套：{ name, projects: [...], settings: {...} }
- PostgreSQL 需要拆表 + JOIN，MongoDB 直接嵌套

### 11. MongoDB CRUD vs Supabase
- Supabase 链式调用（.select().eq().order()），MongoDB 条件对象（.find({ name: "大鹅" })）
- MongoDB 更新必须加 $set：不加 = 整个文档替换（危险），加了 = 只改指定字段
- 嵌套查询：$gte（≥）、$lt（<）、$in（在列表里）、$ne（≠）、$or（或）

### 12. MongoDB vs PostgreSQL 选型
- 结构固定、需要 JOIN → PostgreSQL；结构灵活、字段多变 → MongoDB
- 用户/订单/金融 → PostgreSQL；商品/日志/配置 → MongoDB

### 13. Redis 缓存基础
- Redis 存在内存里，比硬盘快 100~1000 倍
- Redis ≈ 数据库层面的 useMemo（缓存结果避免重复查询）
- 三个基本命令：set（存）、get（取）、set + EX（存+设过期时间）
- 必须设过期时间：防内存爆掉 + 防数据过期不一致
- Redis 只在服务器端（所有用户共享），客户端有自己的 HTTP 缓存

### 14. 缓存三大问题
- 穿透：请求不存在的数据绕过缓存 → 解决：缓存空值
- 击穿：热门数据过期瞬间大量请求打数据库 → 解决：互斥锁只让一个去查
- 雪崩：大量缓存同时过期 → 解决：过期时间加随机值
